WebSocket Pull API de Ducksboard

19 agosto, 2012 Osukaru 20 comentarios

Ducksboard es un panel de control para monitorizar métricas de negocio en tiempo real. Básicamente consiste en uno o varios tablones en los que vamos añadiendo widgets (contadores, gráficos, “timelines”, etc.) que nos muestran información de diferentes servicios de los que hace uso nuestra aplicación web (Google Analytics, Mailchimp, Twitter, Facebook, Teambox, Github y muchos muchos más). Esto mola mucho, porque tenemos todos esos numeritos y dibujitos que tanto les gustan a esa gente que se ocupa de decidir si una aplicación web es o no es una buena inversión reunidos en un mismo sitio. Pero lo que es aún más molón es que Ducksboard nos pone a nuestra entera disposición sus APIs y nos regala una cuenta developer con la que podremos trastear todo lo que queramos.

En total tenemos 4 APIs distintas para jugar:

  • Push API: Nos permite nutrir con datos a los widgets de nuestros tablones.
  • HTTP Pull API: Nos permite recuperar la información de todos nuestros widgets.
  • WebSocket Pull API: Nos permite recuperar la información de nuestros widgets mediante el uso de WebSockets.
  • Dashboard API: Nos permite, entre otras cosas, crear y eliminar tablones y widgets de nuestra cuenta de Ducksboard.

De las dos primeras ya habló @moisesgallego en este vídeo y creó un Bundle para que podamos hacer uso de ellas en nuestros proyectos Symfony2. Yo voy a hablar de la WebSocket Pull API, que es muy sencilla de utilizar una vez que entendemos todos los conceptos que entran en juego.

La nueva especificación WebSocket de HTML5 define un API que establece una conexión persistente entre el cliente y el servidor en la que ambas partes pueden enviar datos en cualquier momento.

Para abrir una conexión con el servidor WebSocket de Ducksboard únicamente tenemos que instanciar la clase WebSocket a través de su constructor:

[javascript]
var socket = new WebSocket(‘wss://ws.ducksboard.com’);
[/javascript]

Podemos comprobar el estado de la conexión mediante la propiedad readyState de nuestro socket

[javascript]
console.log(“Estado de la conexión ” + socket.readyState);
[/javascript]

Que en este caso devolverá el valor 0, que indica que está conectandose. Cuando la conexión se ha establecido se genera un evento onopen y el valor de readyState cambiará a 1:

[javascript]
socket.onopen = function() {
  console.log(“Estado de la conexión ” + this.readyState);
};
[/javascript]

Con esto ya estamos preparados para enviar y recibir mensajes, pero lo cierto es que Ducksboard no nos enviará ningún mensaje hasta que no le hayamos enviado un mensaje de login. El momento adecuado para enviar este mensaje es justamente cuando la conexión se ha establecido correctamente y lo que Ducksboard nos pide que le enviemos en ese mensaje de login es un objeto JSON con las claves message, api_key y timezone en donde el message será el texto “login” (Nota: Si nuestro tablón es público podremos loguearnos usando el access_token y opcionalmente el access_password en lugar de nuestra api_key). Para ello crearemos un objeto Javascript y lo serializaremos a formato JSON:

[javascript]
socket.onopen = function() {
  socket.send(JSON.stringify({
    message: “login”,
    api_key: my_api_key,
    timezone: my_timezone
  }));
};
[/javascript]

Cuando el servidor haya recibido nuestro mensaje nos enviará un mensaje de vuelta confirmándonos nuestro login. Para comprobar este mensaje únicamente tenemos que escuchar el evento onmessage. El mensaje que nos envía Ducksboard es un JSON con la clave message y el valor welcome, por lo que tendremos que deserializarlo para comprobar el mensaje:

[javascript]
socket.onmessage = function(event) {
  var response = JSON.parse(event.data);
  console.log(response.message);
[/javascript]

Ahora Ducksboard ya sabe quienes somos, pero no sabe qué queremos. Hay dos acciones básicas que nos permite realizar Duckasboard: recuperar información de los widgets (fetch) y suscribirnos a un widget para que Ducksboard nos informe cada vez que es actualizado (subscribe).

Depende un poco de lo que queramos hacer, pero llegados a este punto lo que parece más lógico es primero recuperar la información de los widgets que nos interesen para inicializar los gráficos en los que queramos mostrar esa información y una vez que se ha completado esta operación suscribirnos a esos widgets para que nuestros gráficos se actualicen en tiempo real.

Podemos mirar en la documentación de la API el formato que tienen que tener nuestros mensajes “fetch” y “subscribe” y también el que tendrán los mensajes que nos va a devolver el servidor para confirmar nuestras peticiones (“fetched” y “subscribed”), además del mensaje de tipo “data” que nos enviará cada vez que se actualice alguno de nuestros widgets. Nuestro código Javascript podría tener esta pinta:

[javascript]
socket.onmessage = function(event) {
  var response = JSON.parse(event.data);
  switch(response.message) {
    case “welcome”:
      socket.send(JSON.stringify({
        message: “fetch”,
        label: my_label,
        kind: “last”,
        count: 5
      }));
      break;
    case “data”:
      console.log(response)
      break;
    case “fetched”:
      console.log(response);
      socket.send(JSON.stringify({
        message: “subscribe”,
        label: my_label
      }));
      break;
    }
};
[/javascript]
